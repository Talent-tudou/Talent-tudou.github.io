IC笔记2

  • 启动sequence的方法

    • sequence.start()的方式来显示启动
    • default sequence来隐式启动
    • 也可以通过系列宏`uvm_do来启动

  • 面向对象编程的优势

    • 易维护：采用面向对象思想设计的结构，可读性高，由于继承的存在，即使改变需求，那么维护也只是在局部模块，所以维护起来是非常方便和较低成本的。
    • 质量高：在设计时，可重用现有的代码，在以前的项目的领域中已经被测试过的类使系统满足业务需求并具有较高的质量。
    • 效率高：在软件开发时，根据设计的需求对现实世界的事物进行抽象，产生类。使用这样的方法解决问题，接近于日常生活和自然的思考方式，势必提高软件开发的效率和质量。
    • 易拓展：由于继承、封装、多态的特性，自然设计出高内聚、低耦合的系统结构，使得系统更加灵活、更容易拓展，而且成本较低。

  • 事件的触发

    • 用来触发事件的，使用->
    • 用来等待事件的@或者wait

  • 约束的形式

    • 权重约束dist
      • =n，表示每一个取值权重都是n
      • /n，表示每一个取值权重为n/num
    • 条件约束 if else 和 ->(case)
      • if else和正常使用一样的；->通过前面条件满足后可以触发后面事件的发生
    • 范围约束inside
      • inside {[min:max]}；范围操作符，也可以直接使用大于小于符号进行，但是不可以连续使用，如min<wxm<max这使错误的

  • get_next_item和try_next_item

    • get_next_item()是一个阻塞调用，直到存在可供驱动的sequence item为止，并返回指向sequence item的指针。
    • try_next_item()是一个非阻塞调用，如果没有可供驱动的sequence item，则返回空指针。

  • 断言and和intersect的区别

    • and：指的是两个序列具有相同的起始点，终点可以不同
    • intersect：指的是两个序列具有相同的起始点和终点
    • or：指的是 两个只要满足一个就可以
    • throughout：指的是满足前面要求才能执行后面的序列

  • break；continue；return

    • break语句结束整个循环
    • continue立即结束本次循环，继续执行下一次循环
    • return语句会终止函数的执行，并返回函数的返回值（如果有的话）
    • return之后的语句，在function里面是不会执行的

  • 触发器和锁存器的区别

    • 触发器：时钟触发，收到时钟的控制，是由在时钟触发的时候才采样当前的输入，产生输出
    • 锁存器：由电平触发，非同步控制。在使能信号有效时锁存器才相当于时通路，在使能信号无效时锁存器保持输出状态。触发器是由时钟沿触发，同步控制。
    • 锁存器对输入电平敏感，受布线延迟影响较大，很难保证输出没有毛刺产生；触发器则不易产生毛刺

  • 编写测试用例

    • 主要是写sequence，然后再body里面根据要测试的功能要求，写相应的激励，然后再通过ref_model和checker判断功能是否实现了

  • 自动执行脚本

    • tcl脚本，makefile脚本

  • 断言$past的用法

    • 例如让你写一个abcd四个信号在时钟沿处检测，当cd同时为1的时候，在时钟的前两个周期要ab同时为1的断言

  • 关闭约束

    • constraint_mode(0)关闭默认范围的约束块
    • constraint_mode(1)是打开约束
    • 可以用soft关键字修饰特定的约束语句，这样既可以让变量在一般的情况下使用默认值，也可以直接给变量赋默认值外的值

  • 队列的方法

    • insert、delete、push_back和pop_front
    • push插入，pop取出
    • front前边，back后边

  • rand和randc的区别

    • rand修饰符：rand修饰的变量，每次随机时，都在取值范围内随机取一个值，每次取值随机到的概率都是一样的，就像是在掷色子一样。
    • randc修饰符：randc表示周期性随机，即所有的可能值都取过后，才会重复取值

  • 组件之间的通信

    • 单向通信、双向通信和多向通信
    • 单向通信
      • 是指从initiator到target之间的数据流向是单一方向的
    • 双向通信
      • 双向通信的两端也分为initiator和target，但是数据流向在端对端之间是双向的。
    • 多向通信
      • 仍然是两个组件之间的通信，是指initiator和target之间的相同的TLM端口数目超过一个时的处理解决方法

  • blocking阻塞传输方法

    • put()：initiator先生成数据data，同时将该数据送至target。
    • get()：initiator从target获取数据，而target中的数据则应该消耗
    • pee()：initiator从target中获取数据，而target中的数据不消耗，保留

  • 通信管道

    • TLM FIFO：可以进行数据缓存，功能类似于mailbox，不同的地方在于uvm_tlm_fifo提供了各种端口(put、get、peek)用户使用
    • analysis port：一端对多端，用于多个组件同时对一个数据进行处理，如果整个数据是从同一个源的TLM端口发出达到不同组件，则要求该端口能够满足一端到多端，如果数据源端发生变化需要通知跟它关联的多个组件时，我们可以利用软件的设计模式之一观察者模式实现：即广播模式。
    • analysis tlm fifo
      • 由于analysis端口提出了实现了一段到多端的tlm数据传输，而一个新的数据缓存组件类uvm_tlm_analysis_fifo为用户提供了可以搭配uvm_analysis_port端口和uvm_analysis_imp端口和write()函数。
      • uvm_tlm_analysis_fifo类继承于uvm_tlm_fifo，这表明它本身就具有面向单一TLM端口的数据缓存特性，同时该类又有一个uvm_analysis_imp端口analysis_export并且实现了write()函数。

  • request & response通信管道

    • 双向通信端口transport，即通过在target端实现transport()方法可以在一次传输中既发送request又可以接受response。

  • 深拷贝和浅拷贝

    • 浅拷贝可以使用列表自带的copy()函数(list.copy())，或者使用copy()模块的copy函数。深拷贝要使用deepcopy()，所有要from copy import deepcopy
    • 如果拷贝的对象里面的元素只有值，没有引用，那么浅拷贝就够了，都是复制一份原来的对象，在新的对象里面修改值的大小，不会对原来的对象造成影响，两个时完全分开的。
    • 如果里面包含了引用，那么深/浅拷贝就有区别了，复制了之后，改变了新对象的中的引用，原来对象中的引用同样会被修改，两个没有完全分开，而深拷贝就会在新的对象中重新创建一片空间，在复制过来，两个完全分开。

  • public、protected和local

    • 如果没有指明访问类型，那么成员的默认类型时public的，子类和外部均时可以访问的
    • 如果指明了访问类型时protected，那么就只有该类的或者是其子类可以访问的，外部是不可以访问的
    • 如果是local的话，那么只有该类是可以访问的，子类和外部都是不可以访问的

  • UVM验证方法学的理解

    • 刚开始接触的时候，我简单的认为就是SV的一个封装，将我们在搭建测试平台过程中的一些重复的，重要的工作进行的封装，从而使得我们可以快速的搭建一个可重用性高的测试平台，因此当时我还觉得就是一个库而已。
    • 随着学习的深入，学习了UVM中的各种机制和模型的构造和相互之间的关系后，我觉得UVM方法学对于使用何种语言其实不重要，重要的是它的思想，比如在UVM中有sequence机制，以往如果我们要使用SV进行TB搭建时候，我们一般会采用driver类进行数据的产生，转换，发送或者generator和driver两个类进行上述的功能，这种方式的可重用性就很低，而且代码很臃肿，但是在UVM中我们使用sequence，sequencer、driver、sequence_item拆开，相互独立而又相互联系，因此我们只需要关注于每一个需要做的工作就可以了，可重用性高。除此之外，UVM方法学中的很多思想都是来源于经验，来源于生活，而不在乎是何种语言。

  • UVM的验证环境结构，各个组件之间的关系

    • UVM测试平台由object和component组成的，其中component搭建了TB的一个树形结构，其基本包含了driver、monitor、sequencer、agent、scoreboard、model、env、test和top；然后object一般是包含了sequence_item，sequence、config和一些其他需要的类。各个组件相互独立，又通过TLM事务级传输进行通信，除此之外，DUT与driver和monitor又通过interface进行连接，实现驱动和采集，最后在top层进行例化，调用test进行测试。

  • UVM中的机制

    • uvm中由很多的机制，factory机制，field_automation机制、phase机制，打印机制、sequence机制，config_db机制等，这些机制使得我们在搭建的uvm平台可以有很好的重用性和使得我们平台运行有序稳定。

    • phase机制：主要是使得uvm的运行仿真层次化，使得各种例化先后次序正确。uvm的phase机制有9个，外加12个小phase。主要的phase有build phase、connect phase、run phase、report phase、final phase等，其中除了run phase是task，其他的都是function，然后build phase和final phase都是自顶向下运行，其余都是自底向上运行的。run phase和12个小的phase(reset phase、configure phase、main phase、shutdown phase)是并行运行的，这12个小phase主要是进一步将run phase中的事务划分到不同的phase中进行了。注意：run phase和12个小的phase最好不要同时用。

    • 从运行上看，9个phase的执行顺序是顺序执行的，不同组件中的同一个phase执行有顺序，build phase是自顶向下执行，只有同一个phase全部执行完毕才会执行下一个phase

  • phase中的domain

    • domain是用来组织不同组件的，实现独立运行的概率。默认情况下，uvm的9个phase属于common_domain，12个小的phase属于uvm_domain。例如，如果我们有两个driver类，默认情况下，两个driver类中的复位phase和main phase必须同时执行，但是我们可以设置两个driver属于不同的domain，这样两个driver就是独立运行的了，相当于是处于不同的时钟域（只针对12个小的phase有效）

  • run_phase和main_phase之间的关系

    • run_phase和main_phase（动态运行）都是task phase，且是并行运行的，后者称为动态运行(run-time)的phase。
    • 如果想要执行一些耗时的代码，那么要在此phase下任意一个component中至少提起一次objection，这个结论只适用于12个run-time的phase。对于run_phase则是不适用的，由于run_phase与动态运行的phase是并行运行的，如果12个动态运行的phase有objection被挂起，那么run_phase就不需要raise_objection了。

  • main_phase要如何到reset_phase

    • 在main phase执行的过程中，突然遇到了reset信号被置起，可以用jump()实现从main_phase到reset_phase的跳转。

  • uvm组件的通信方式tlm的接口分类和用法以及peek和get的差异

    • uvm中采用事务级传输机制进行组件之间的通信，可以大大提高仿真的速度和使得我们简化组件间的数据传输，简化工作，TLM独立于组件之外，降低组件之间的依赖关系。UVM接口i主要是由port，export和imp；驱动这些接口的方式有put，get，peek，transport，analysis等。
    • 其中peek是查看端口内部的数据事务但是不删除，get是获取后立即删除。我们一般会先使用peek获取数据，但是不删除（保证put端不会立马又发送一个数据），处理完毕后在使用get删除。
    • imp只能是终点接口，transport表示双向通信，analysis可以连接多个imp（类似于广播）

  • analysis port是否可以不连接或者连接多个import

    • 都是可以的，analysis port是类似于广播的，其可以同时对多个impo进行事务的通信，只需要在每个imp端口声明write()函数即可。对比put，get，peek port，他们都只能进行一对一传输，且也必须申明对应的函数如put()、get()、peek()、can_put()/do_put()等。FIFO是可以不用申明操作函数的，其内部封装了很多的通信端口，如analysis_export等，我们只需要将端口与其相连就可以实现通信了。

  • sequence和item

    • item是基于uvm_object类的，这表明了它具备了uvm核心基类的数据操作方法，如copy()、clone()、compare()、record()等。
    • item对象的声明应该起始于sequence的body方法，而后经历了随机化并穿过了sequencer到达driver，直到被drvier消化之后它的声明一般来讲才会结束。
    • item和sequence的关系：一个sequence可以包含一些有序组织起来的item实例，考虑到item在创建之后需要被随机化，sequence在声明时也需要预留一些可以供外部随机化的变量，这些随机变量一部分是用来通过层级传递约束来最终控制item对象的随机变量的，一部分时用来对item对象之间加以组织和时序控制的。

  • sequence的分类

    • 扁平类（flat sequence）：这一类往往只用来组织更小的粒度，即item实例构成的组织。
    • 层次类（hierarchical sequence）：这一类是由更高层的sequence用来组织底层的sequence，进而让这些sequence或者按照顺序方式，或者按照并行方式，挂载到同一个sequencer上。
    • 虚拟类（virtual sequence）：这一类则是最终控制整个测试场景的方式，鉴于整个环境中往往存在不同种类的sequencer和其对应的sequence，我们需要一个虚拟的sequence来协调顶层的测试场景。之所以称之为virtual sequence，是因为该序列本身不会固定挂载于某一个sequencer类型上面，而是将其内部不同类型sequence最终挂载到不同的sequencer上面。这也是virtual sequencer不同于hierarchical sequence的最大一点。
