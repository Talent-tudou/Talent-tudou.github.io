发表于2023-06-25|更新于2023-06-25|IC

IC笔记3

  • sequence和sequencer的关系

    • sequence机制用于产生激励，它是uvm中最重要的机制之一。sequence机制有两个部分组成：sequence和sequencer。
    • 在整个验证平台中sequence处于一个比较特殊的位置。sequence不属于验证平台的任何一个部分，但是它由于sequencer有着十分密切的关系。
    • 只有在sequencer的帮助下，sequence产生的transaction最终才可以送到driver；同样地，sequencer只有在sequence出现的情况下才能体现出价值，如果没有sequence，sequencer几乎没有任何的作用。
    • sequence是一个uvm_object，而sequencer是一个uvm_component。sequence和transaction一样的是有声明周期的。它的声明周期比transaction更长一点，其内部的transaction全部发送完毕后，它的声明周期也就结束了。

  • sequencer的仲裁特性(set_arbitration)以及锁定机制(lock和grab)

    • 仲裁特性

      • uvm_sequencer::set_arbitration(UVM_SEQ_ARB_TYPE val)函数来设置仲裁模式，模式有几种选择
        • UVM_SEQ_ARB_FIFO：默认模式，来自于sequence的发送请求，按照fifo先进先出的方式依次授权，和优先级没有关系
        • UVM_SEQ_ARB_WEIGHTED：不同sequence的发送请求，将按照它们的优先级权重随机授权
        • UVM_SEQ_ARB_RANDOM：不同的请求会被随机授权，而无视它们的优先级和到达顺序
        • UVM_SEQ_ARB_STRICT_FIFO：不同的请求，会按照它们的优先级以及抵达顺序来依次授权，因此与优先级和抵达事件都有关
        • UVM_SEQ_ARB_STRICT_RANDOM：不同的请求，会按照它们的最高优先级随机授权，和抵达时间无关
        • UVM_SEQ_ARB_USER：用于可以自定义仲裁方法user，priority，arbitration来裁定哪一个sequence的请求被优先授权

    • 锁定机制

      • 有两种锁定机制，lock()和grab()

        • lock和unlocak这一对方法可以为sequence提供排外的访问权限，但前提条件是该sequence已经按照得到了sequencer的授权，一旦获得了授权，就不需要担心权限被收回了，只有该sequence主动解锁（unlock）它的sequencer才可以释放这一锁定的权限。lock是一种阻塞的任务，只有获得了权限，它才会返回。

        • grab与ungrab也可以为sequence提供排外的访问权限，而且它只需要在sequencer下一个授权受其就可以无条件的获取全新。与lock方法相比，grab方法无视同一时刻内发起传送请求的其他sequence，而唯一可以阻止它的只有已经预先获取授权的其他lock或者grab的sequence。

        • 这里需要注意的是，“解铃还须系铃人”，如果sequence使用了lock或者是grab方法，必须在sequence结束前条用unlock或者是ungrab方法来释放权限，否则sequencer会进入死锁状态而无法继续为其余的sequence授权。

  • virtual sequence和virtual sequencer中virtual的含义

    • virtual含义就是sequencer不需要传递item，也不会与其他的driver相连，其只是一个取协调各个sequencer的中央路由器。通过virtual sequencer我们可以实现多个agent的多个sequencer，它们的sequence的调度和重用。virtual sequence可以组织不同sequencer的sequence群落。

    • 见笔记2

  • 如何在driver中使用interface

    • interface如果不进行virtual声明的话是不能直接使用在driver中的，会报错，因为interface声明是一个实际的物理接口，一个在driver中使用virtual interface进行申明接口，然后通过config_db进行接口参数的传递，这样我们可以从上层组件中获得虚拟的interface接口进行处理。
    • config_db传递只能传递virtual接口，即interface的句柄，否则传递的就是一个实际的物理接口，这在driver中是不能实现的，且这样的话不同组件中的接口一一对应一个物理接口，那么这样的操作是没有意义的。

  • callback机制

    • callback机制是为了提高TB的可重用性，其还可以进行特殊激励的产生等，与factory类似，两者可以有机结合使用。与factory不同之处在于callback的类还是原先的类，只是内部的callback函数变了，而factory是产生一个新的拓展类进行替换。
      • uvm组件中内嵌callback函数或者是任务
      • 定义一个常见的uvm_callback class
      • 从uvm_callback空壳类拓展uvm_callback类
      • 在验证环境中创建并登记uvm_callback

  • field_automation机制

    • field_automation机制：可以自动实现copy、compare、print等一些函数，当使用uvm_field系列宏注册之后，可以直接调用以上函数，而无需自己定义。这极大的简化了验证平台的搭建，尤其是简化了driver和monitor，提高了效率。

  • objection机制

    • uvm中通过objection机制来控制验证平台的关闭，需要在drop_objection之前先raise_objection，验证在进入到某一个phase时候，uvm会先收集此phase提出的所有objection，并且实时检测所有objection是否已经被撤销了，当发现所有都已经撤销后，那么就会关闭此phase，开始进入下一个phase。当所有的phase都执行完毕后，就会调用$finish来讲整个验证平台关掉。如果uvm发现此phase没有提起任何objection，那么将会直接跳到下一个phase中。
    • uvm的设计哲学就是全部由sequence来控制激励生成，因此一般情况下只是在sequence中控制objection。另外还需要注意的是，raise_objection语句必须再main_phase中第一个消耗仿真时间语句之前。

  • config_db机制

    • 主要作用就是使得TB的可配置性更高，更加灵活

    • 传递interface，通过传递virtual interface使得driver和monitor可以和DUT相连，并驱动接口和采集接口信号

    • 传递单一变量参数，如int，string，enum等，这些主要就是为了配置某些循环次数，id号是多少等

    • objection类的传递，这种主要是当配置参数多的时候，我们可以将其封装到一个类里面，同时包含了处理这些参数的相关方法，这样传递起来就比较方便，也不易出错。

    • config_db主要是由四个参数组成的：第一个参数是父的根parent，第二个参数是对应的组件，第三个是传递时候的名字，第四个是变量名。

    •   uvm_config_db#(virtual interface)::set(uvm_root::get(), "uvm_test_top.c1", "vif", vif)
  uvm_config_db#(virtual interface)::get(this, "", "vif", vif);

  • uvm_component是如何组织运行的

    • component之间通过new函数创建指定parent参数指定子关系，通过这种方法来讲TB形成一个树形结构。uvm中运行时通过phase机制进行层次化仿真的。从组件来看各个组件并行运行，从phase上看是串行运行的，有层次化。phase机制的9个phase是串行运行的，不同组件中的同一个phase都运行完毕后才能进入下一个phase运行。同一个phase在不同组件中的运行也是由一定顺序的。

  • uvm如何启动一个sequence

    • 启动sequence有很多的方法：常用的方法有使用default sequence进行调用的，其会将sequence与sequencer进行绑定，当driver请求获得req时，sequencer就会调用对应的sequence去运行body函数，从而产生req。
    • 除此之外，还可以使用start函数进行，其参数主要就是对应的需要绑定的sequencer和该类上层的sequence，如此，就可以实现启动seque的功能。
    • 注意：一般仿真开始结束会在sequence中raise和drop其objection。

  • 为什么要RAL（寄存器）

    • 首先，我们要了解寄存器对于设计的重要性，其实模块间交互的窗口，我们可以通过读寄存器值取观察模块的运行状态，通过写寄存器取控制模块的配置和功能改变。
    • 然后，为什么我们需要RAL呢？由于前面的寄存器的重要性，我们可以直到，如果我们不能先保证我们寄存器的读写正确，那么就不用谈后续的DUT是否正确了，因此，寄存器的验证是排在首要位置的。
    • 那么我们应该用什么方法取读写和验证寄存器呢？采用RAL寄存器模型取测试验证，是目前最成功的方法，寄存器模型独立于TB之外，我们可以搭建一个测试寄存器的agent，去通过前门或者是后门访问去控制DUT的寄存器，使得DUT按照我们的要求去运行。
    • 除此之外，uvm中还内建了很多RAL的sequence，用于帮助我们去检查寄存器，还有一些其他的类和变量去帮助我们塔尖，以提高RAL的可重用性和便捷性还有更全的覆盖率。

  • 前门/后门访问

    • 前门访问：顾名思义就是指在寄存器模型上做读写操作，最终会通过总线UVC来现实总线上的物理时序访问，因此是真实地物理操作

    • 后门访问：指地是利用UVM_DPI(uvm_hdl_read()、uvm_hdl_deposit())，将寄存器的操作直接作用到DUT内寄存器变量，而不通过物理总线访问。

    • 前门访问在使用时需要将path设置为UVM_FRONTDOOR

    • 在进行后门访问时候，用户首先需要确保寄存器模型在建立时候，是否将各个寄存器映射到了DUT一侧的HDL路径，使用add_hdl_path

    • 前门访问 后门访问
      通过总线协议访问，需要耗时。总线访问结束时，前门访问才结束。 通过UVM DPI关联硬件寄存器路径，直接读取或修改硬件，不消耗仿真时间，零时刻响应。
      一般只能按字(word)读写，无法直接读写寄存器域。 可以对寄存器或者寄存器域直接读写。
      依靠总线检测来对寄存器模型做预测。 依靠auto predition方式对寄存器模型做预测。
      反映了正确的时序关系。 不受硬件时序控制，对硬件做的后门访问可能发生时序冲突。
      不受总线时序功能影响。 通过总线协议可以有效捕捉总线错误，从而验证总线访问路径。

    • 从上面的差别可以看出来，后门访问相较于前门访问更加快捷一些，但如果单纯依赖后门访问也是不行的

    • 实际上，利用寄存器模型的前门访问和后门访问混合方式，对寄存器验证的完备性更有帮助。

  • 后门访问的路径怎么配置

    • 通过uvm_reg_block::add_hdl_path()，将寄存器模型关联到了DUT一端，而通过uvm_reg::add_hdl_path_slice完成了将寄存器模型各个寄存器成员与HDL一侧的地址映射。
    • 另外，寄存器模型的build函数最后以lock_model函数结尾，该函数的功能是结束地址映射关系，并且保证模型不会被其他用户修改。
    • 在寄存器模型中完成了HDL路径映射之后，我们才可以利用uvm_reg或者uvm_reg_sequecen自带的方法进行后门访问，后门访问也有几种方法提供：
      • uvm_reg::read/write()在调用读方法的时候需要注明UVM_BACKDOOR的访问方式
      • uvm_reg_sequence::read_reg()/write_reg()在使用的时候需要注明UVM_BACKDOOR的访问方式
      • uvm_reg::peek()/poke()两个方法，也分别对应了读取寄存器（peek）和修改寄存器（poke)两种操作。而用户无需指定访问方式，因为这两个方法就是针对后门的

  • 如果寄存器的地址不匹配怎么测试

    • 在通过前门配置寄存器A之后，在通过后门访问判断HDL地址映射的寄存器A变量值是否改变了，最后通过前门访问来读取寄存器A的值

  • 寄存器模型的常规方法

    • mirror、desired、actual
    • 我们在使用寄存器模型的时候，除了利用它的寄存器信息，也会利用它来跟踪寄存器的信息。寄存器有很多的域，每一个域都有两个值
    • 寄存器模型中的每一个寄存器，都应该有两个值，一个是镜像值（mirror value），一个是期望值（desired value）
    • 期望值是先利用寄存器模型修改软件的值，而后利用该值更新硬件的值；镜像值是表示当前硬件的已知状态值
    • 镜像值往往由模型预测给出，即在前门访问时通过观察总线或者时在后门访问的时候通过自动预测等方式来给出镜像值
    • 镜像值可能域硬件的实际值不一样

  • prediction机制

    • UVM提供了两种用来跟踪寄存器值的方法，我们将其分为自动预测（auto prediction）和显示预测（explicit）。

    • 如果用户想要使用自动预测的方法，还需要调用函数uvm_reg_map::set_auto_predict()。

    • 两种预测方式的显著差别在于，显示预测对寄存器模型预测更为准确，我们可以通过下面对两种模式的分析得到具体的原因

      • 如果用户没有在环境中继承独立的predictor，而是利用寄存器的操作来自动记录每一次寄存器的读写操作，并在后台自动调用predict()方法的话，这种方式被称为自动预测。

      • 这种方式时简单有效的，然后需要注意的是，如果出现了一些其他的sequence直接在总线层面上对寄存器进行操作（跳过了寄存器级别的write/read操作，或者是通过其它的总线来访问寄存器等这些额外的情况，都无法自动得到寄存器的镜像值和预期值）。

      • 更为可靠的一种方式是在物理总线上通过监视器来捕捉总线事务，并将捕捉到的事务传递给外部例化的predictor，该predictor由UVM参数化类uvm_reg_predictor例化并集成在顶层环境种。

      • 在集成的过程种，需要将adapter与map的句柄一并传递给predictor，同时将monitor采集的事务通过analysis port接入到predictor一侧。

      • 这种集成关系可以使得，monitor一旦捕捉到了有效事务，会发送给predictor，再由其利用adapter的桥接方法，实现事务信息转换，并将转化的寄存器模型有关信息更新到map种。

      • 默认情况下，系统将采用显示预测的方式，这就要求集成到环境中的总线UVC monitor需要具备捕捉事务的功能和对应的analysis port，以便于同predictor相连接。
