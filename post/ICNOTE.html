IC验证笔记

  • fork join | fork join_any | fork join_none | wait fork | disable fork

    • fork join：内部begin end块并行运行，直到所有线程运行完毕才会进入下一个阶段。
    • fork join_any：内部的begin end块也是并行运行的，直到有任意一个begin end块运行结束就可以进入到下一个阶段了。
    • fork join_none：内部的begin end块也是并行运行的，无需等待直接进行到下一个阶段。
    • wait fork(name)：会引起调用进程的阻塞，直到它里面的所有子线程结束，一般是用来保证所有的子线程都运行结束了。
    • disable fork(name)：用来终止调用进程的所有活跃的进程，以及进程的所有的子进程。

  • 多线程的同步调度

    • mailbox：主要是用于两个线程之间的数据通信，通过put函数和get函数以及peek函数进行数据的发送和获取。
    • event事件：主要是用于两个线程之间的同步运行，通过事件的触发和事件等待实现两个线程之间的运行同步。使用@(event)或者是wait(event.trigger)进行等待，->进行触发。
    • semaphore：主要是用于对于资源访问的一个交互，通过key的获取和返回实现一个线程对资源的访问，使用put和get函数获取返回key，一次可以多个。

  • task和function

    • function可以调用另一个function，但是不能调用任务（按照道理是可以调用没有耗时的task的）。
    • task可以调用function，也可以调用function。
    • 函数总是在仿真的0时刻就开始执行了，任务可以在任何非零时刻执行。
    • 函数一定不能包含任何延迟、事件或者是时序控制声明语句，任务就可以包含延迟、事件或者是时序控制语句。
    • 函数==至少==是一个输入变量，可以有多个输入变量，任务可以没有或者多个输入（input）、输出（output）和双向（inout）变量。
    • 函数只能是返回一个值，函数不能有输出output或者是inout，任务不返回任何值，任务可以通过输出output和双向inout变量来传递多个值。

  • interface和clocking blocking

    • interface是一组接口，用于对信号进行一个封装，捆扎起来。如果像verilog中对各个信号进行连接，每一层我们都需要对接口信号进行定义，若信号过多，很容易出现人为错误，而且后期的可重用性不高。因此使用interface接口进行连接，不仅可以简化代码，而且可以提高代码的可重用性，除此之外interface内部还提供了一些功能，用于测试平台于DUT之间的同步和避免竞争。
    • clocking blocking：在interface内部我们可以定义clocking blocking块，可以使得信号保持同步，对于接口的采样vrbg和驱动有详细的设置操作，从而避免TB和DUT的接口竞争，减少我们由于信号竞争导致的错误。采样提前，驱动落后，保证信号不会出现竞争。

  • OPP（面向对象）的特性

    • ==封装==：通过将一些数据的和使用这些数据的方法封装在一个集合里面，成为一个类。
    • ==继承==：允许通过现在有的类去得到一个新的类，且其可以共享现有类的属性和方法。现有类叫基类或者父类，新类叫做派生类或者子类。
    • ==多态==：得到拓展类后，有时候我们会使用基类（父类）的句柄指向一个子类的对象，这个时候是如何准确的判断我们想要调用的方法呢？通过对类中方法进行virtual声明，这样当使用父类的句柄指向子类对象的时候，方法就会根据对象去识别，自动的调用子类的方法了，而不是父类的方法。而父类和子类中方法有着同样的名字但是能够准确的调用，就叫做多态。

  • UVM的工厂机制

    • Factory机制也叫做工厂机制，其存在的意义就是为了能够方便的替换TB中的实例或者是已经注册的类型。一般而言，搭建完了TB之后，我们如果需要对TB进行更改配置或者更改相关类的信息，我们可以通过使用factory机制进行覆盖，达到替换的效果，从而大大提高TB的可重用性和灵活性。
    • 使用factory机制，要先进行：
      • 将类注册到factory的表中。
      • 创建对象的时候，使用对应的语句：type_id::create()。
      • 编写相应的类对基类进行覆盖。

  • UVM的启动

    • 在导入uvm_pkg文件时候，会自动创建==uvm_root==所例化的对象==uvm_top==，uvm顶层的类会提供run_test()方法充当uvm世界的核心角色，通过uvm_top调用run_test()方法。
    • 在环境中输入run_test来启动uvm验证平台，run_test语句会创建一个my_case0的实例，得到正确的test_name，依次执行uvm_test容器中的各个component组件中的phase机制。
      • build_phase（==自顶向下==构建uvm树）
      • connect_phase（==自底向上==连接各个组件）
      • end_of_elaboration_phase
      • start_of_simulation_phase
      • run_phase()：objection机制，通过start启动sequence（每个sequence中都又一个body任务，当一个sequence启动后，会自动执行sequence中的body），等到sequence发送完毕之后落下objection，结束run_phase（uvm_objection提供component和sequence共享的计数器，当所有参与到objection机制中的组件都落下了的狮虎，计数器count才会清零，才满足run_phase退出的条件）

  • Interface的传递

    • 传递单一的变量，传递配置对象
    • 虽然SV可以通过层次化的interface的索引来完成索引，但是这种传递方式不利于软件环境的封装和复用。通过使用uvm_config_db配置机制来传递接口，可以将接口的传递和获取彻底分开。
    • 接口传递从硬件世界到uvm环境可以通过uvm_config_db来实现，但是在实现的过程中，应该要注意：
      • 接口的传递应该发生在run_test()之前，这保证了在进入==build_phase==之前，virtual interface就已经被传递到了uvm_config_db中
      • 用户应该把interface和virtual interface 区分开来，在传递的过程中的类型应该为virtual interface，即实际接口的句柄
    • 在配置单一变量的时候：
      • 在各个test中，可以在build_phase阶段对底层组件的各个变量加以配置，进而在环境例化之前完成配置，使得环境可以按照预期运行。
    • 传递配置对象到环境：
      • 在test配置中，需要配置的参数不只是数量多，可能还分属于不同的组件。对这么多层次的变量做出类似上边的单一变量传递，需要很多的代码，容易出错而且不易复用。
      • 如果整合各个组件中的变量，将其放置在一个uvm_object中，再对中心化的配置对象进行传递，将有利于整体环境的修改维护，提升代码的复用性。

  • UVM的优势（好处）

    • UVM其实就是SV的一个封装，将我们在搭建测试平台过程中的一些重复性和重要的工作进行封装，从而使我们可以快速的搭建一个需要的测试平台，并且可重复性还高。但UVM不仅仅是封装。

  • ref类型

    • reg参数类型是引用（参考c/c++中的指针传递）。
    • 向子程序传递数组时应尽量使用reg获取最佳的性能，如果不希望子程序改变变量的值，可以使用const ref类型。
    • 在任务里可以修改变量而且修改结果对调用这个变量的函数随时可见（因为传递的指针，指向的就是真实的数据，如果是传递到的数值，进入function/task的时候会进行一次复制）

  • component和object的区别，item是什么？

    • uvm中component也是object派生出来的，不过相比于object而言， component有很多其没有的属性，例如phase机制和树形结构等。在uvm中，不仅仅需要component这种较为复杂的类，进行TB的层次化搭建，也需要object这种基础类进行TB的==事务搭建==和一些环境配置等。
    • item是object

  • uvm的树形结构

      flowchart TD
      A["uvm_test_top"] --> |my_case0| B["env"]
      B --> C{agent}
      B --> D(scoreboard)
      B --> E(model)
      B --> F(monitor)
      C --> G(sequencer)
      C --> H(driver)

  • UVM验证环境的组成

    • sequencer：负责将数据传送给driver
    • driver：负责数据的发送；driver有时钟/时序的概念
    • agent：其实只是简单的把driver， monitor和sequencer封装在一起
    • agent：对应的是物理接口协议，不同的接口协议对应不同的agent，一个平台上面一般会有多个agent
    • env：相当于是一个特大的容器，将所有的成员都包含进去
